- Explain the feature in detail

#### Getting Started

1. Fork the repository
2. Clone your fork locally
3. Create a new branch for your feature/fix
   ```bash
   git checkout -b feature/your-feature-name
   ```
4. Make your changes
5. Test thoroughly
6. Commit with clear messages
7. Push to your fork
8. Open a Pull Request

#### Code Style Guidelines

**HTML**
- Use semantic HTML5 elements
- Include proper meta tags
- Maintain consistent indentation (2 spaces)
- Add ARIA labels for accessibility

**CSS**
- Keep styles in `<style>` blocks or separate files
- Use consistent naming conventions
- Comment complex styles
- Maintain the neon/pirate theme aesthetic
- Use CSS variables for colors when possible
- Keep mobile-first responsive design

**JavaScript**
- Use ES6+ features where appropriate
- Comment complex logic
- Keep functions small and focused
- Use meaningful variable names
- Add error handling

**Colors to Use**
- Cyan: #00ffff, #0ff, #19f7d2
- Yellow: #ffe14b, #fff200
- Neon Green: #39ff14
- Dark backgrounds: #000, #111, #0a1230
- Use text-shadow for glowing effects

#### Testing Your Changes

Before submitting:
1. Test on multiple browsers (Chrome, Firefox, Safari, Edge)
2. Test on mobile devices
3. Verify all links work
4. Check console for errors
5. Test keyboard navigation
6. Validate HTML/CSS

#### Commit Message Format

Use clear, descriptive commit messages:
```
Add: New chatbot response for F1 queries
Fix: Roulette wheel animation bug on mobile
Update: Improve village map responsiveness
Docs: Add contributing guidelines
```

### What We're Looking For

Priority areas for contribution:
- **Chatbot Intelligence** - More conversation patterns
- **Game Features** - New games or improvements
- **Accessibility** - ARIA labels, keyboard navigation
- **Performance** - Optimization and loading times
- **Mobile UX** - Better mobile experience
- **Documentation** - Code comments, guides
- **Bug Fixes** - Fixing existing issues

### What to Avoid

Please don't:
- Remove existing features without discussion
- Change the core theme/aesthetic drastically
- Add large dependencies unnecessarily
- Submit untested code
- Use copyrighted content without permission

## Development Workflow

1. **Branch Naming**
   - `feature/` - New features
   - `fix/` - Bug fixes
   - `docs/` - Documentation updates
   - `refactor/` - Code refactoring

2. **Pull Request Process**
   - Fill out the PR template completely
   - Link related issues
   - Include screenshots for UI changes
   - Wait for review
   - Address feedback promptly

3. **Code Review**
   - All PRs require review before merging
   - Be open to feedback
   - Respond to comments
   - Make requested changes
